# 失控重复处理：看门狗与计数 Tag 机制

> **配套文档**
> - 理论背景：[《主论文》](主论文.md) §3（逻辑死锁）与 §4（探针 + 选择性介入）
> - 机制分类与术语来源：[《机制分类》](机制分类.md)
> - 代理层透明性约束：[《代理_透明性》](代理_透明性.md)

本文档记录失控重复处理的具体实现。它是 [《主论文》](主论文.md) 中所论证的 (B) 类反射机制的一个具体实例——属于"核心部分"，与"代理部分"严格解耦。

## 一、定位

主论点已经在主论文里讲过：失控重复属于一个**逻辑死锁**——模型在故障态下注意力已经塌掉、无法自检；所以"是否处于故障态"必须由外部判定。判定的关键是要拿到一份**注意力的存活证明**，而这份证明必须是模型只有在注意力良好时才能稳定提供的东西。

本文档要回答的就是：这份"存活证明"具体长什么样、怎么拿到。

## 二、核心思路：让模型周期性"喂狗"

让 AI 在输出过程中，**每 K 个单词输出一个特殊计数 Tag**。代理程序持续读取这些 Tag，依据它们是否按顺序、按节奏出现，判定模型当下的注意力是否仍然在场。

> **没有计数 → 注意力涣散。计数错乱 → 注意力涣散。**

这是直接的存活证明：维持一个递增、按节奏出现的计数，要求模型有完整的注意力；注意力一旦塌掉，计数立刻就会缺失或错乱。

## 三、强版本：固定周期计数

最直接的实现：

```
... <up to K words> [TAG:1]
... <next K words>  [TAG:2]
... <next K words>  [TAG:3]
...
```

规则：

- Tag 必须严格按整数递增
- 每 K 个词必须出现一个 Tag
- 缺失、跳号、乱序 → 触发介入

代理读到的 Tag 流就是模型注意力的实时心跳。这一版的代价是**常态化的 token 开销**：每段输出都要带 Tag。但好处是无论是否进入失控态，监控都在线。

## 四、弱版本：按需触发计数

如果不想常态付计数代价，可以做一个按需版本：

1. 代理常态只跑一个**廉价的重复检测器**（字符串/相似度匹配，属于 (A) 信号机制层的事——便宜、不消耗模型注意力）。
2. 一旦检测到首尾相连的典型重复模式，**才检查计数 Tag 是否在场**。
3. 模型预先被告知一条规则：**如果你打算输出重复内容，每个重复单元后面必须带计数 Tag**——这是它在主动重复时证明自己注意力良好的方式。
4. 由此分出两条路径：
   - 检测到重复 + 计数 Tag 完整正确 → 可控重复（"说 10 次你好"），不动它。
   - 检测到重复 + 计数缺失或错乱 → 失控重复，触发截断 + 注入新信息。

弱版本的关键在于规则的**不对称性**：

- 主动（可控）重复时，模型注意力在场，能稳定遵守"每个重复单元带 Tag"这条规则。
- 失控重复时，模型注意力已经塌掉，遵守规则所需的能力本身已经失效——所以 Tag 不会出现，或者出现得乱七八糟。

正是这种不对称，把"内部状态从外部读不到"这件事在外部映射成了"Tag 是否符合规则"——两个表面相同的重复（可控 vs 失控）就被这一道筛子分开了。

## 五、为什么这恰好是"AI 看门狗"

计算机系统里的看门狗 (watchdog timer) 工作原理是：被监控的进程必须周期性"喂狗"——发送心跳信号；心跳超时则看门狗判定进程已挂，自动重启它。

这一套结构跟计数 Tag 机制是一一对应的：

| 看门狗 | 计数 Tag 机制 |
| --- | --- |
| 心跳信号 | 计数 Tag |
| 喂狗动作 | 输出 Tag |
| 心跳超时 / 错乱 | Tag 缺失 / 错乱 |
| 重启进程 | 截断输出 + 注入新信息（重新分布注意力） |

所以这件事可以一句话说清：**计数 Tag 机制是 AI 界的看门狗。** 它工作在模型的注意力之外，模型本身既不知道自己在被监督，也无法绕过这套监督。

## 六、与代理层的接口

计数 Tag 是机制层的内部协议，不能泄漏到用户层。这意味着：

- 用户输入恰好包含 Tag 明文时，代理必须在送给模型之前转义。
- 模型若要输出 Tag 明文（作为普通文本，而非作为机制本身的计数），必须输出转义版本，由代理在送回用户之前逆转义还原。
- Function Call、JSON 模式、流式输出等所有涉及 Tag 出现的场景都要被覆盖。

这部分由代理层负责，详见 [《代理_透明性》](代理_透明性.md)。

## 七、待定的实现细节

以下是几个需要进一步选定的工程参数，记录在这里待后续展开：

- **K 的选择**：太小 → token 开销过大；太大 → 检测延迟过长。可能需要按场景动态调整。
- **Tag 语法**：实际使用的 Tag 形式（例如 `[#WD:N]` / `<wd:N>` / 其他）需要选一个不与常见用户内容冲突、又便于转义的形式。
- **流式判定时机**：在 streaming 输出场景下，介入触发应该在哪一帧、滞后多少 tokens——这是一个延迟与误判率之间的权衡。
- **介入策略的内容**：截断之后注入"什么新信息"——是固定模板、还是基于上下文动态合成？两种各有取舍，应单独讨论。
