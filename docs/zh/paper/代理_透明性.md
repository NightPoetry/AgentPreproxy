# 代理层：特殊 Tag 的透明性处理

> **配套文档**
> - 计数 Tag 机制本身：[《实现_看门狗与计数Tag》](实现_看门狗与计数Tag.md)
> - 整体原理：[《主论文》](主论文.md)（特别是 §5 关于代理层的论述）
> - 机制分类与术语来源：[《机制分类》](机制分类.md)

本文档记录代理层对计数 Tag（以及其他机制层内部协议 Tag）的透明性处理。它属于"代理部分"，与"核心部分"严格解耦——核心部分负责机制本身（比如计数 Tag 怎么用），代理部分只保证机制对使用者完全不可见。

## 一、透明性原则

**使用者不应该感觉到机制层的存在。** 他输入什么、得到什么，应当跟没有这套代理时一致。计数 Tag 是机制层的内部协议，必须严格留在机制层内部，不向外泄漏。

形式化地讲，这转化成两个**对称**的转义约束：

1. **用户输入侧**：用户输入恰好包含 Tag 明文时，代理在送给模型之前必须将其转义。
2. **模型输出侧**：模型如果要输出 Tag 明文（作为普通文本，而非机制本身的计数），必须输出转义版本；代理在送回用户之前再做逆转义还原。

只要这两条对称约束都满足，整条往返路径上 Tag 的明文都只在"代理 ↔ 模型"这条内部链路上出现，外部使用者永远看不到。

## 二、模型必须被告知转义规则

代理层有一项前置工作：在系统提示词或注入指令中，明确告诉模型——

> **"如果你需要在输出里出现这个 Tag 的明文（作为普通文本的一部分），请输出转义版本而不是明文。"**

这条约束的优先级必须很高。原因是：模型一旦输出未转义的 Tag 明文，代理就无法区分两件事：

- 这是机制层的计数 Tag（应当被代理消费、不能转给用户）
- 这是用户希望看到的文本中正常包含的 Tag 字符串（应当原样转给用户）

两种意图混在同一个序列里，机制就废了。所以这条规则不是"建议"，是"硬约束"——它必须像普通的输出格式约束一样被模型严格遵守。

## 三、转义必须覆盖所有输入/输出模式

特殊 Tag 的转义不能只在自由文本模式下生效。**任何 Tag 可能出现的输入或输出通道都要被覆盖**，至少包括：

- 普通文本输出
- 流式输出 (streaming)
- Function Call 的参数体
- JSON 模式（结构化输出）
- 工具调用 (tool use) 的参数与返回值
- 多模态输入中的文本片段

任何一个通道遗漏了转义，整套透明性就破了——使用者那一边就会看到机制内部协议的明文，或者机制层自己被用户输入污染。

## 四、JSON 模式的具体处理

JSON 模式值得单独说，因为它涉及对用户传入的 schema 做改写。

**典型流程**：

1. 用户传入一个 JSON schema（描述他想要的输出结构）。
2. 代理检查这个 schema：如果某个字段的值有可能包含 Tag 明文，代理在送给模型之前**改写 schema 或注入字段级约束**——要求该字段中所有 Tag 字符出现都以转义形式输出。
3. 模型按照修改后的 schema 输出，得到的是带转义 Tag 的 JSON。
4. 代理在把结果送回用户之前，对相关字段做**逆转义**——把转义 Tag 还原为明文。
5. 用户收到的 JSON 跟没有代理时是一样的：结构没变，字段值是干净的明文。

**关键点**：

- 用户视角下，他传入的 schema 没有被改、他收到的 JSON 没有变形——透明性成立。
- 模型视角下，它收到的 schema 与它实际输出的 JSON 都遵守"Tag 必须转义"的规则——机制层的协议被维护。
- 这一套"用户的 schema 不变 → 模型实际看到的是改过的 schema → 输出经过逆转义还原"的流程，是 JSON 模式下保持透明性的核心。

Function Call、Tool Use 的参数体处理是同一个套路：代理对这些通道里的 Tag 字符做对称的转义/逆转义。

## 五、流式输出的处理

流式输出 (streaming) 比较特殊，因为模型是一个 token 一个 token 流出来的，代理必须做边界识别：

- 如果当前 token 流是一个完整的计数 Tag → 代理消费、不向用户转发。
- 如果当前 token 流是转义后的 Tag 明文 → 代理逆转义后转发给用户。
- 如果当前 token 流是其他正常文本 → 直接转发。

实际中常见的麻烦是：Tag 可能跨多个 token，代理要做缓冲与边界识别——这一段属于工程实现细节，不在本文档的原理范围内，留待具体实现时展开。

## 六、总结

整条透明性约束可以被一句话概括：

> **不管使用者用什么模式、传什么内容，他都不应该感觉到代理的存在。** 输入端转义出去，输出端逆转义回来；闭环里有计数 Tag 在跑，闭环外什么都看不见。

只有这一条满足，机制层（计数 Tag 看门狗等）才能既不打扰用户、又能正常工作。透明性是代理层存在的核心价值——这也是为什么"代理 + 核心"这种双层结构必须严格解耦的原因之一：核心管"机制做什么"，代理管"机制怎么不被看见"。
