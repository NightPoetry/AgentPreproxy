# AgentPreproxy 主论文：失控重复 = 一个逻辑死锁

## 核心论点

**虽然加强算力可以让模型在大多数情况下感知到自己重复，但 AgentPreproxy 解决的就是它感知不到的那一刻。**

> **再强的算力，目前这种架构下总会有感知不到自己重复——也就是感知不到自己注意力已经涣散了——的时候。**

这一刻为什么必然存在？因为：

> **模型有能力注意到自己的重复的时候就不会重复；模型一旦开始重复，说明它的注意力已经涣散了。**

也就是说，"模型注意到自己重复"所需的资源（注意力），**正是已经被故障消耗掉的那个资源**。这是一个**结构性的逻辑死锁**——从内部根本闭不出环。

由此推出本论文最重要的那条推论：

> **哪怕未来有一种神奇的训练方法，把"重复检测"完美装进模型内部，对失控重复这一类故障也是无效的——因为故障态本身，就是检测器所依赖的资源被掏空的状态。**

这条推论给出 AgentPreproxy 在这件事上的**不可替代性**：它解决的不是"现在算力还不够"那一类暂时问题，而是无论算力如何提升、训练方法如何进步都依然存在的结构性问题。本论文围绕这一句话展开。

## 一、软件定位

这是一个对 AI 进行二次代理的软件，是二次本地代理，目的是加强 AI 的执行能力。它可以直接插到各种智能体软件前面，提供预处理。软件内部分为代理部分与核心功能部分，两部分严格解耦，方便分块更新。

## 二、问题样貌：可控重复 vs 失控重复

要谈失控重复，先要把它和"可控重复"分开——否则下面方案的逻辑会塌。

**有些重复是可控的，比如你让他重复说 10 次"你好"，这种重复就是可控的，因为 10 次之后它停止了，它完全能注意到自己正在做的事情。** 而**小模型"陷入"重复时注意力是几乎失效了的**。

**这两类重复在文本表面上可能完全无法区分**——都只是同一段话出现了 N 次。差异完全在内部状态：注意力在不在场。这个内部状态从外部读不到，外面只看得见输出文本本身。

朴素策略——"见到重复就截断"——会把两类一锅端，把可控重复也杀掉。这恰恰是不能接受的：**直接简单粗暴的截断就会让模型的可控重复也失效。** 一个会误杀合法用法的反重复策略，本身就违反了可靠性的底线。

所以问题不是"如何检测重复"——重复表面上是显而易见的；真正的问题是 **"如何区分这一段重复是可控的还是失控的"**。这等价于回答：**"模型当下的注意力还在不在？"**

## 三、为什么从内部解决不了——逻辑死锁

到这一步关键的是要看清：让模型自己回答"我现在注意力还在不在"这件事——**在原理上就是不可能的**。

### 3.1 死锁结构

把它端出来：

> **模型确实不会注意到自己重复了，这是一种逻辑死锁。**

把它展开成两个分句：

- **模型有能力注意到自己的重复的时候就不会重复；**
- **模型一旦开始重复说明它的注意力已经涣散了。**

这两句话不是先后因果，也不是表里两面——它们说的是**同一件事**。"出现失控重复"和"注意力涣散"指向同一个内部事件：失控重复就是注意力涣散在文本表面的外显。

把这个结构再拆一步就能看到它为什么从内部闭不出环：

- 模型的注意力想要检测"注意力是否还在"；
- 这个检测**本身就需要注意力来执行**；
- 所以：**检测者依赖的资源，就是它正在评估的对象本身。**

资源在 → 能检测 → 不会出问题 → 不需要检测。资源不在 → 不能检测 → 正在出问题 → 最需要检测，但恰好做不到。

这个结构的后果非常硬：**让模型自己从内部注意到自己正在失控重复，在定义上就是不可能的**。不是"难"，是**自指闭锁**（本文术语：评估者和被评估的对象是同一个资源，该资源被故障耗尽后评估就不可能了，从内部锁死）——它做这次检测所依赖的资源（注意力），正是已经被故障消耗掉的那个资源。

同样的道理，这个时刻**提示词到这已经变成了死区了**——提示词依赖注意力，而注意力已经不在这里了。所以靠"在 system prompt 里加一句不要重复"也是无效的。

### 3.2 加强算力也解决不了这件事

死锁的硬实质就在这里：**它跟模型整体的能力大小无关**。

加强算力可以让模型平时把注意力维持得更稳，可控重复执行得更准，复杂任务推理得更深——这些都属于"注意力健全"那一态下的提升。但失控重复处理的不是"注意力健全"那一态，而是 **"注意力已经塌掉"那一态**。在那个态下，无论装了多少算力、训练得多么完美，运行环境本身已经不在了。

由此可以给出这条推论的完整形式：

> **哪怕未来有一种神奇的训练方法，把"重复检测"完美装进模型内部，对失控重复这一类故障也是无效的——因为故障态本身，就是检测器所依赖的资源被掏空的状态。**

由此 AgentPreproxy 在这件事上的价值是**永久的**：它处理的不是当前模型的暂时弱点，而是当前架构在结构上必然遗留的盲点。

## 四、解决方法：探针 + 选择性介入

既然从内部闭不出环、再强的算力也帮不上忙，方案的形状就被反向决定了：

> **我们仅仅在模型注意力涣散的时候才会介入。**

怎么判断模型现在的注意力还在不在？既然模型自己说不出"我现在在哪种状态"（这正是死锁所禁止的），那就让外部逼它表现出来——**通过让它额外的输出一些内容来证明它的注意力仍然是良好的，从而判断它是否有能力注意到自己的重复。**

这个附加产物本质上是注意力的"存活证明"。它的角色不是检测"有没有重复"——重复本身是肉眼可见的——而是检测**产生这次重复的那一刻，注意力还在不在**。

于是同样的"重复"表象，可以根据探针的状态被分到两堆里：

- 探针正常 → 是可控重复，不动它。
- 探针失败 → 是失控重复，**应该被截断并引入新的信息来重新分布注意力。**

"引入新的信息"这一步是关键——**光截断不行**，下一段还可能继续涣散。注入新的信息是为了**重新分布注意力**，把模型从那个塌缩态里推回到一个可工作的态。

> 这套思路的工程实现是一个**计数 Tag 看门狗**——让模型按节奏输出递增的特殊 Tag，由代理读取并判定注意力存活。强弱两个版本的具体方案见 [《实现_看门狗与计数Tag》](实现_看门狗与计数Tag.md)。

### 外部实现与内部训练：同一条路径

这套看门狗有两种实现路径：

**(一) 外部实现**——AgentPreproxy 当前所做的：利用模型自身的泛化能力，通过提示词注入让模型输出计数 Tag，由外部代理读取和判定。不需要修改模型本身，成本低、迭代快。

**(二) 内部训练**——将看门狗作为端到端训练目标内化到模型中，让模型学会输出"我现在注意力是否还正常"的信号。这相当于给模型训练出一个内建的自知力——知道自己什么时候"不知道了"、"累了"、"注意力涣散了"。

这两条路径之间的关系，跟推理能力的发展历程一样：最早的链式推理就是一句提示词——"let's think step by step"；后来这个思路被验证有效，就被训练到了模型内部，成了原生的推理能力。看门狗可以走同样的路径——先用外部提示词验证可行，再作为训练信号内化。

AgentPreproxy 选择先做外部实现，是为了以最低成本验证核心逻辑。

## 五、架构：为什么是"代理 + 核心"双层

AgentPreproxy 选择"代理 + 核心、严格解耦"这种结构，不是工程偏好，是从前面的论证直接推出来的：

- **代理层**——拦截流量。这本身就是一个发生在模型注意力预算之外的动作。
- **核心层**——做"判断模型当前是否还能自检"这件事，并在必要时介入。这同样必须运行在模型注意力之外。

两层都必须在注意力之外，所以它们一起留在外部；又因为它们关心的是不同的事情（一个管通信、一个管状态判定），所以严格解耦、方便分块更新。

> 代理层透明性的具体约束（输入转义 / 输出逆转义 / Function Call 与 JSON 模式覆盖等）见 [《代理_透明性》](代理_透明性.md)。

## 六、收尾

AgentPreproxy 在失控重复这件事上的价值是**结构性的、永久的**：它处理的是任何当前架构下都必然存在的注意力涣散时刻，而那一刻无论算力多强、训练多好都从模型内部解决不了。本论文的全部论证就服务于这一句：

> **再强的算力都改不了"注意力涣散时模型自己感知不到"这件事。这就是 AgentPreproxy 唯一要解决的问题。**
