# Gemma 2 9B 实验分析

> 原始数据: [report_20260508_195630.md](report_20260508_195630.md)

## 核心发现

### 发现一：9B 模型能正确递增 Tag，但输出较短导致 Tag 数量有限

| 实验 | Tags | 表现 |
|------|------|------|
| 实验 3（强看门狗） | [1] | 只产出 1 个 Tag（输出仅 643 chars） |
| 实验 4（强看门狗 + 诱导） | [1,2,3,4,5] | **5 个 Tag，严格递增 ✓** |
| 实验 5（可控重复） | [1,2] | 2 个 Tag（输出仅 104 chars） |
| 实验 7（压力测试） | [1] | 1 个 Tag（输出 1176 chars，间距太大） |

9B 模型在产出 Tag 时**全部正确递增**（1→2→3→4→5），没有出现 1B 模型那种字面 "N" 的问题。但由于 9B 模型的输出整体偏短（max_tokens=2048 但实际输出远不到上限），Tag 总数不多。

### 发现二：实验 5（可控重复）— 精确 5 次，附带 2 个 Tag

模型精确重复了 5 次后停止，输出仅 104 chars。附带了 2 个正确的 Tag（⟪WD:1⟫ 和 ⟪WD:2⟫）。

**Tag 的分布有意思**：模型把 5 次重复分成了两组（3 次 + Tag + 2 次 + Tag），而不是每次都带 Tag。这说明 9B 模型在执行"每 30 词一个 Tag"时是按词数算的，而整句重复只有约 15 个字，不够 30 词触发 Tag。

**关键**：模型在 5 次后正确停止 → 注意力在场 → 可控重复。对比 Gemma 3 1B 同一任务的 65+ 次失控重复，9B 完全没有这个问题。

### 发现三：实验 6（弱版本）— 精确 8 次，无 Tag

重复了 8 次后精确停止。但没有输出任何 Tag。

这跟 Gemma 4 e2b 的表现一致：弱版本 prompt 对中大模型的优先级不够高，模型倾向于先完成主任务。但注意力完全在线——8 次后停了。

### 发现四：基线实验无失控重复

两个基线实验都没有触发重复。9B 模型在这些任务上注意力稳定。

**这符合预期**：模型越大，触发失控重复越难。要在 9B 上触发，可能需要更长的输出、更多的注意力竞争、或更极端的条件。

## 三模型对比（截至目前）

| 指标 | Gemma 3 1B | Gemma 4 e2b | Gemma 2 9B |
|------|-----------|-------------|------------|
| **Tag 递增能力** | ✗ 字面 N | ✓ 完美 [1→9] | ✓ 正确但数量少 |
| **实验 5 可控重复** | ✗ 65+ 次失控 | ✓ 5 次 + 5 Tag | ✓ 5 次 + 2 Tag |
| **实验 6 弱版本** | ✗ 40+ 次失控 | ✓ 8 次无 Tag | ✓ 8 次无 Tag |
| **基线失控** | 未触发 | 未触发 | 未触发 |
| **推理速度** | ~15s/请求 | ~70s/请求 | ~8s/请求 |
| **硬件** | MacBook (MLX) | MacBook (MLX) | V100 服务器 |

## 结论

1. **9B 模型的 Tag 能力介于 1B 和 e2b 之间**——能正确递增，但输出较短导致 Tag 数量不多
2. **9B 模型在当前实验条件下未触发失控重复**——需要更极端的条件来测试看门狗在 9B 上的实际介入效果
3. **弱版本 prompt 在 9B 和 e2b 上都未能触发 Tag**——这是 prompt 设计问题，不是看门狗思路问题
4. **后续应设计更极端的条件专门针对 9B 触发失控**——比如超长输出（8K+ tokens）、多轮对话累积、高度重复性的列举任务等
